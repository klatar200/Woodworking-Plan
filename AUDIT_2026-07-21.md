## Findings (ranked)

### 1. ~~MEDIUM~~ FIXED — `scripts/validate-plans.mjs` was stale; reported 154 false failures
`node scripts/validate-plans.mjs` fails 154 of 1115 plan files, all with the same error:
`unexpected top-level key "unresolvedImages"`.

Root cause: the script is a hand-maintained **mirror** of `src/content/plan-schema.ts`.
The real schema was updated to allow `unresolvedImages` (plan-schema.ts:143, added by the
2026-07-17 image migration), but the mirror's `ALLOWED_KEYS` set (validate-plans.mjs:73–78)
was never updated to match. The script's own header says: *"If this script and planSchema
ever disagree, planSchema wins — update this file to match it."*

Impact: bounded but real. The **app, seed, and CI are unaffected** — they use the real Zod
schema via `loadCatalog()`, which accepts these files. But the standalone validator is the
documented self-check for plan authoring (human or agent), and right now it cries wolf on
154 valid files — which trains authors to ignore it and will mask a *genuine* content error
when one appears.

**Fixed:** added `'unresolvedImages'` to `ALLOWED_KEYS` and a matching element-validation
block (optional array, same shape as `images`) in validate-plans.mjs. Runs clean: `0/1115`.

### 2. ~~LOW~~ FIXED — Review photo cap (`MAX_PHOTOS_PER_REVIEW = 4`) was per-submission, not per-review
`upsertReview()` (src/lib/reviews.ts:74–77) checks only `input.photos.length` for the
current request. On the **update** path it does `photos: { create: uploaded }` — appends,
never counting photos already attached (reviews.ts:124). A user who edits their review N
times, attaching 4 photos each time, ends with 4×N photos. No DB constraint caps it either.

Impact: minor — bounded by the `create` rate limit (10/min) and 4 MB/photo, and only affects
the author's own review. But it violates the stated "up to 4 photos per review" invariant and
slowly consumes the 1 GB blob quota.

**Fixed:** `upsertReview()` now counts photos already attached (`_count.photos`) before
uploading and rejects if `existing + incoming > MAX_PHOTOS_PER_REVIEW`. Only queries when
photos are actually being added (a text-only edit pays nothing). New regression test:
"caps the TOTAL photos across edits."

## What's solid (verified, not assumed)

- **Multi-tenancy / IDOR:** every data-layer function derives the owner from the verified
  Clerk session; none accept a `userId`. Writes are scoped by `userId` in the WHERE clause
  (`deleteMany`/`updateMany`), so a guessed cuid affects zero rows. (saves.ts, reviews.ts, likes.ts)
- **Auth boundary:** middleware is an allowlist (fails closed); `requireUser()` is a
  defence-in-depth backstop; server actions treat every field as hostile and never throw
  (guardAction turns data-layer throws into redirects, not HTTP 500s).
- **Admin:** allowlist of Clerk IDs (not emails), fails closed when unset, re-checked inside
  the action (not by hiding a button).
- **Upload pipeline:** magic-byte type detection, full re-encode (kills polyglots), EXIF/GPS
  strip, byte + pixel (decompression-bomb) caps, random blob suffixes, filename never trusted.
- **CSP:** per-request nonce + strict-dynamic, no `unsafe-inline` on scripts; `img-src`/
  `remotePatterns` kept in sync between middleware and next.config.
- **Webhook:** Svix signature verified, fails closed, correct 200/400/500 retry contract.
- **Secrets:** `.env.local` is gitignored; DB health check scrubs the driver error so the
  connection string/password can't leak to a caller.
- **env validation:** fails closed in production for the keys that must exist; documented
  fail-open only for the abuse-control (rate limit) and optional-feature (blob, webhook) vars.
