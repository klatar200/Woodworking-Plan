## P0 — Critical (broken product, or next routine action destroys content)

### 1. `/plans/[slug]/build` does not exist — "Start building" 404s everywhere

The page was **never committed in any commit** (`git log --all` on
`src/app/plans/[slug]/build*` is empty), but everything that points at it shipped:

| Pointer | Location |
|---|---|
| "Start building →" CTA (the only path to steps once JS mounts) | `src/components/instructions-disclosure.tsx:47` |
| Plan page renders that CTA and hides inline steps after hydration | `src/app/plans/[slug]/page.tsx:417` |
| Service worker pre-caches `/plans/${slug}/build` on save | `src/components/service-worker.tsx:90` |
| Offline library download includes it | `src/lib/offline-urls.ts:41` |

Verified live: `GET /plans/cedar-raised-garden-bed/build` → **404** (screenshot taken).
For every JS user the step-by-step instructions — the core content — are unreachable,
online and offline. The SW silently skips the 404 while still reporting
`DOWNLOAD_COMPLETE`. `StepWalker` (`src/components/step-walker.tsx`) is consequently
dead code in production, imported only by tests — which is why 547 green tests never
noticed.

Almost certainly the page exists **untracked on Keagan's Windows machine** (the
2026-07-16 session notes describe it browser-verified). **Fix:** recover and commit
`src/app/plans/[slug]/build/page.tsx`, or rebuild it: a public page (already covered by
the `/plans(.*)` allowlist entry and cacheable), rendering `PlanSteps` + `StepWalker`
full-page, **no ViewLogger** (Sprint 19: the plan page already logged the view). Add a
test that the route module exists so a pointer-without-page can never ship again.

### 2. `scripts/apply-step-tags.mjs` now destroys the 61 new plans' step tags on re-run

The script iterates **every** file in `content/plans/` and rebuilds each step from its
`TAGS` table only — it never reads the step's existing `tools`/`materials`:

```js
function emitStep(step, tag) {
  const tools = (tag?.tools ?? []).filter((t) => t);   // TAGS[slug] is undefined
  const mats  = (tag?.materials ?? []).filter((m) => m); // for all 61 new plans
```

`TAGS` covers exactly the original 24 slugs. The 61 new plans carry tags **inline** in
their JSON, so one re-run — the workflow CLAUDE.md still documents as "editing the tags
= editing that script + re-running it" — silently strips every step tag from 61 plans.
The loader would still pass (tags are optional), so nothing catches it before a seed.
The script's own header ("SAFE TO RE-RUN … a second run is a no-op") is now false.

**Fix:** make the script skip files whose slug is not in `TAGS` (one-line guard), or
retire it entirely by folding the 24 plans' tags into their JSON (making plan JSON the
single source of truth for all 85) and deleting the script + updating CLAUDE.md. The
second option removes the split-brain permanently and is recommended.

## P1 — Major

### 3. Dark mode: every ghost/plain `<button>` is invisible

`globals.css` deliberately excludes Tailwind preflight and has **no `button` reset**;
the Sprint 29 `btn`/`btnGhost`/`btnDanger`/`btnLiked` constants (`src/lib/ui.ts:42-54`)
set border + text color but **no background**. So `<button>`s keep the UA default
`ButtonFace` fill. Measured in the running app (dark theme): the sort "Apply" button is
`background: rgb(239,239,239)` with `color: rgb(240,236,228)` — **≈1.0:1 contrast,
literally invisible** (screenshot taken). Affects every `<button>` using those
variants in dark: sort Apply, filter Apply/like/shopping-list/save toggles, walker
Prev/Next, review submit. `<a>` elements with the same classes render transparent, so
light mode also has a subtle button-vs-link fill inconsistency nobody noticed. The
Sprint 32 "AA audit" audited tokens, not UA defaults; `colorScheme` is `normal` (unset).

**Fix:** add a button reset to the `globals.css` base block
(`button { background: transparent; font: inherit; color: inherit; }` — or
`bg-transparent` on `btnBase`), and set `color-scheme: light dark` keyed off the theme
so UA form internals follow. Re-check the dark screenshots after.

### 4. 12 new plans ship an empty `cutList` — and the home page now overclaims

built-in-bookcases, camper-napkin-holder, catch-all-tray, dachshund-cd-holder,
diy-buffet-table, diy-wooden-tool-box, modern-nesting-tables,
nesting-cutting-boards-and-egg-holder, ombre-cutting-board, spice-drawer-organizer,
toy-sink-running-water, wooden-centerpiece-tray. All 24 originals have one. The UI
degrades safely (tab/optimizer/print section are conditional), but the catalog
sub-heading — **"85 plans — every one with a full cut list, material list, and cost
estimate."** (`src/app/page.tsx`) — is now false for 12 of them, and the Sprint 15
board optimizer is dead for furniture-scale builds like built-in-bookcases.

**Fix:** author the 12 cut lists (preferred for the furniture-scale ones at minimum),
and soften the sub-heading until then.

### 5. The 61 new plans need Keagan's provenance sign-off before they reach production

`research/reference-plans.json` (committed, public repo) is an explicit scrape ledger —
per-plan `source_url`/`source_site` (ana-white.com ×14, myoutdoorplans, construct101,
…), `extraction_method`, and a self-declared caveat: "Source sites hold standard
copyright… Do not copy source instructional prose." Against that standard:
**53/61 titles are copied exactly** from the sources (including branded collection
names: "Homestead …" is Ana White's brand; slugs `charlie-nightstand`,
`matts-media-tower` keep source-branded names in public URLs), and **15 plans contain
1–3 step sentences verbatim** from the ledger's `original_steps` (worst:
cutting-board-storage-rack 3/6 steps, jacobs-ladder-toy 3/5, plant-stand-magazine-rack
3/5, toy-sink-running-water 3/7). Descriptions spot-checked clean. Source sites 403'd
verification of the ledger's "written fresh" claim, so treat it as unverified.

All 61 are `published: true` and go live on the next production seed. Per
`CLAUDE.md` §1 and §5 this is a legal/branding call — **it is Keagan's, explicitly,
before any prod seed**. Recommended engineering support either way: retitle the
branded plans, re-slug the two named ones (slug changes are cheap pre-launch), rewrite
the 15 plans' verbatim step sentences, and move `research/` out of the public repo
(it is a public confession ledger with outbound links).

### 6. New content is invisible in production until someone seeds

Standing deploy rule: schema flows on deploy, **data does not**. The 61 plans, 5 new
tools, and any step-tag edits exist only in git + whichever database has been seeded.
Production still serves the 24-plan catalog until `db:seed` runs against prod
(procedure in `DEPLOYMENT.md`) — which should happen only **after** #5 is decided and
#2 is fixed.

## P2 — Minor defects

7. **Mobile drawer doesn't close on query-string navigation**
   (`src/components/mobile-nav.tsx:26-28`): `useEffect(…, [pathname])` — tapping
   "Home" from a filtered catalog (`/?tools=…`) leaves the drawer open over the new
   results. Join `useSearchParams()` into the dependency.
8. **Tailwind same-property conflict in the drawer install item**
   (`src/components/site-header.tsx:140`): `gap-0` loses to `drawerLink`'s
   `gap-[0.5rem]` (fixed source order — the documented Sprint 29 gotcha). Use `gap-0!`.
9. **Print token reset omits the six functional colors**
   (`src/app/globals.css:77-92`): printing the boards page from dark mode renders
   `.impossible-part` warnings in washed-out light red (`--err: #e8756c`) — the one
   warning the buying list must not soften. Add the functional colors to the print
   reset. (`/plans/[slug]/print` is safe — it forces `color:#000`.)
10. **`install-prompt.tsx:34-50` comment overstates its guarantee**: the
    `beforeinstallprompt` listener attaches post-hydration; if Chrome fires the event
    first, no install affordance appears that load. Pre-existing limitation — fix the
    comment, or capture in a tiny inline script stamped before hydration.
11. **Dead print rule**: `.shopping-list-cta` (`globals.css:1341`) matches nothing —
    the buttons now use `no-print`. Delete the rule.
12. **12 plans have 1–2 untagged steps**; three are real misses (simple-bat-house
    step 2 cuts grooves with no saw tag; ombre-cutting-board step 2 uses finishing oil;
    plant-stand-magazine-rack step 5 sands/finishes). Tag them (in the plan JSON —
    see #2).
13. **Boards page vs. plan intent** (`/plans/cedar-raised-garden-bed/boards`): the
    optimizer's fixed 1″ squaring allowance makes the plan's two legitimately-96″
    "full 8-ft boards, uncut" parts impossible at the default 8-ft setting, and the
    headline "Buy 2 boards" excludes them. The warning is loud (per the Sprint 15
    rule), but the default view contradicts the plan's own cut-list note. Consider an
    "uncut/full-length part" flag or a per-plan default board length.
14. **Thin step bodies** in a few new plans (e.g. plant-stand-magazine-rack step 5 is
    27 chars) — below the catalog's editorial standard.
15. **No new plan appears in any learning path** — all 5 paths reference original-24
    slugs only. Content gap, not a defect.
